## 📊 V2 Architecture Decision Record

**Why SQLModel over raw SQLAlchemy?**
- Pydantic v2 validation + SQLAlchemy ORM in one model
- Type safety end-to-end
- Easy migration to Pydantic v2 native types

**Why Redis + not just DB?**
- Hot cache for portfolio (5s TTL) — prevents DB thrashing on monitor
- Pub/sub for real-time events between agents
- Distributed locking — THE solution for race conditions
- Rate limiting on external APIs

**Why atomic transactions per operation?**
- Position open = create position + deduct balance + log trade (all or nothing)
- Position close = update position + calc PnL + update portfolio + log trade (all or nothing)
- No partial state ever — the corruption bug is impossible

**Why manual confirmation for TIER_1?**
- Highest quality signals need human oversight before real capital
- TIER_2/3 can auto-execute for testing/paper trading
- Gradual autonomy path: manual → semi-auto → full-auto

**Why event-driven?**
- Scanner emits `token_discovered` → Validator listens
- Validator emits `validated` → Risk Engine listens  
- Risk Engine emits `risk_assessed` → Executor listens
- Executor emits `position_opened` → Monitor listens
- Decoupled, scalable, testable

**Why Prometheus + Grafana?**
- Real metrics: scan latency, validation pass rate, position PnL, win rate
- Health dashboards showing all agents
- Alerts on agent down, circuit breaker triggered, disk full

**Why Docker Compose?**
- One command: `docker-compose up`
- PostgreSQL + Redis + all agents + observability stack
- Production-ready, reproducible, scalable
